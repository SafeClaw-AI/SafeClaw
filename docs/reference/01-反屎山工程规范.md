# 反屎山工程规范(通用版)

> **摘要**: 本规范的唯一目标: 从源头消灭屎山,确保代码可维护、可演进。人类定方向拍板,执行者(AI 或开发者)负责编码与质量。规则分四级 — `[门禁]` 自动拦截、`[自检]` 交付前必查、`[人工决策]` 人类审批、`[流程]` 操作规程。增量代码零容忍; 存量结构性债务走白名单限期清零,长期合理例外走例外登记; 敏感信息/空 catch 等即时高风险项发现即阻断,永不延期。  
> **闭环状态**: 本版本经 12 轮迭代定稿。裁决链唯一、放行入口两本账(例外登记+债务白名单)、增量零容忍、跨语言通用。可直接落地。

**优先级**: 本规范优先级高于所有业务开发需求。违反任意红线的输出直接作废,整改至全项通过后方可交付。  
**适用范围**: 任何软件项目,不限开发模式。当项目采用 AI 主导开发(代码主要由 AI 产出)时,第九章自动生效。  
**增量与存量**: 本规范对**增量代码强制执行**,对**存量代码采用债务白名单(限期清零) + 例外登记(长期合理例外)双轨机制**(详见第四章)。  
**度量口径**: 全文"非空行"统一定义为 **去除空白行后的物理行数**(含代码、注释、docstring,不含纯空白行)。  
**检测分级**:
- `[门禁]` — 硬性自动化拦截(CI / pre-commit / selfcheck),不可绕行
- `[自检]` — 执行者每次交付前必须自行检查并整改(AI 主导项目由 AI 执行,非 AI 项目由开发者执行),人类可抽查
- `[人工决策]` — 必须由人类拍板(架构变更、需求边界、例外审批等)
- `[流程]` — 团队/个人必须遵守的操作规程,由执行者(人或 AI)自觉执行

> **全局优先级**: 安全 > 正确性 > 可维护性 > 可读性 > 性能 > 美观。规则冲突时按此顺序裁决。  
> **兜底原则**: 规则冲突时先按全局优先级裁决; 仍无法判定则默认选择更严格的解释,并提示人类确认。冲突影响范围 ≥ 模块级时,AI 必须停下请求人类确认,不得自行裁决。

## 快速上手: 10 条最小可执行清单

新项目或首次落地时,优先执行以下 10 条,其余章节按需渐进:

1. 单函数非空行 ≤80,超限必须拆分 `[门禁]`
2. 单文件非空行 ≤1,200,超限按职责拆文件 `[门禁]`
3. 连续空行 ≤2,空行占比 ≤25% `[门禁]`
4. 禁止空 catch/except 吞异常 `[门禁]`
5. 禁止无主 TODO `[门禁]`
6. 编译/语法检查 100% 通过 `[门禁]`
7. 自动生成文件必须经 formatter 再写入 `[门禁]`
8. 新增代码必须 100% 通过门禁,存量结构性债务走白名单限期清零,长期合理例外走例外登记 `[流程]`
9. 重构前必须先有行为快照/测试兜底 `[流程]`
10. 并行开发必须通过分支/worktree 隔离,提交前校验合并冲突 `[流程]`

## 一、项目基础锁定(源头控风险)

以下边界由人类设定,执行者必须严格遵守,不得自行突破:

1. **目录锁定** `[人工决策]`: 仅可在人工审批定版的目录结构内操作,禁止私自新增/修改/重命名目录。一类文件唯一存放路径。
2. **边界锁定** `[人工决策]`: 仅可修改人工指定的业务模块,禁止触碰架构核心文件、公共基础组件、全局配置、依赖清单。仅使用白名单内的依赖,禁止私自新增。
3. **规划锁定** `[人工决策]`: 仅可在审批通过的需求边界内开发,禁止超范围开发、私自新增功能、修改对外接口定义。

> **执行要点**: 遇到边界模糊时,停下来向人类确认,不得自行决定"应该可以"。

## 二、仓库卫生

### 2.1 根目录准入
根目录仅允许: 运行入口、构建清单(`package.json` / `pyproject.toml` / `Cargo.toml` / `Makefile` 等)、锁文件(`package-lock.json` / `Cargo.lock` 等)、顶层配置(`.gitignore` / `.editorconfig` 等)、`README.md`,以及项目显式约定的例外。其余全部按类别入子目录。

### 2.2 目录分类原则
- **先分类,再落盘; 任务结束前必须收口**
- 正式文档: `docs/`
- 源码: `src/` 或项目约定的源码目录
- 测试: `tests/` 或与源码同目录的 `__tests__/`
- 工具/脚本: `tools/` 或 `scripts/`
- 临时验证产物: `tmp/`,不放长期留痕
- 历史/闲置但需保留: `archive/` 或 `temp/parked/`

### 2.3 命名规则
| 对象 | 规则 | 禁止 |
|------|------|------|
| 文档 | 明确语义,必要时加数字前缀 `01-...` | `最终版`、`临时版`、`new2`、`test1` |
| 方案 | 显式写出状态: `Trial`、`Target`、`草稿` | 模糊标题 |
| 临时快照 | `<来源>-snapshot-YYYYMMDD` | 无日期、无来源 |
| 函数/方法 | 动作+业务域+操作对象+限定场景 `[自检]` | 名称缺少业务对象或场景限定时判定泛化(如裸 `handle_data()`); 若为领域正式术语(如 `DataMigrator`)且与职责一致则不判 |
| 类/组件 | 业务域+职责+类型 `[自检]` | 与文件名不匹配的模糊命名 |
| 常量/枚举 | 业务域_含义_场景(全大写下划线) `[自检]` | 魔法值、硬编码 |

> **命名正反例**:  
> 好: `query_order_status_by_user_id()` / `PaymentRetryService` / `MAX_SESSION_IDLE_SECONDS`  
> 坏: `handle_data()` / `MyService` / `MAGIC_NUM`

### 2.4 文件与测试对应
- 文件名必须与内部核心导出名一致
- 测试文件要么与源码同目录,要么放 `tests/` 且与 `src/` 结构 1:1 对应; 命名固定为 `源文件名.test.后缀` 或 `test_源文件名.后缀`

### 2.5 交付前收口清单
- [ ] 新增文件是否放对目录
- [ ] 临时文件是否已迁走
- [ ] 历史材料是否归档
- [ ] 生成物是否在 `tmp/` 或 `.gitignore`
- [ ] 文档索引是否同步更新

## 三、编码红线(量化,可自检)

### 3.1 体量红线

| 维度 | 标准值 | 极限值 | 超限处理 | 检测 |
|------|--------|--------|----------|------|
| 单函数非空行 | ≤50 | ≤80 | 必须拆分 | `[门禁]` AST 扫描 |
| 单类非空行(有类语言适用) | ≤400 | ≤500 | 必须拆分 | `[门禁]` AST 扫描 |
| 单文件非空行 | ≤800 | ≤1,200 | 按职责拆文件 | `[门禁]` 行数统计 |
| 测试文件非空行 | ≤1,500 | ≤2,000 | 按维度拆测试文件 | `[门禁]` 行数统计 |
| 连续空行 | ≤1 | ≤2 | 格式化修复 | `[门禁]` 行扫描 |
| 空行占比 | ≤20% | ≤25% | 格式化修复 | `[门禁]` 行数统计 |

### 3.2 复杂度红线
- 分支/循环嵌套 ≤3 层,禁止箭头型代码 `[门禁]` 嵌套深度检测
- 单函数圈复杂度 ≤10(核心业务 ≤7) `[门禁]` radon / flake8-cognitive-complexity / eslint-plugin-complexity / SonarQube
- 参数个数 ≤5(超过用对象/结构体封装) `[自检]` 最低标准: 同类型连续参数 ≥3 个即应封装。豁免: UI 组件 Props、只读 DTO/Record 不受此限

> **高复杂度修复示例**: 多层 if-else 分支 → 提取为策略字典/分派表; 多层循环 → 拆成独立函数逐层处理; 长参数列表 → 封装为配置对象。

### 3.3 冗余禁令
- 重复代码 ≥3 行必须抽离公共函数,绝对禁止复制粘贴开发 `[自检]` 最低标准: 同一逻辑出现在两处以上即应抽离
- 所有业务常量/环境配置必须用枚举/配置文件管理,禁止魔法值硬编码 `[自检]` 最低标准: 含义不自明、重复出现、或跨函数复用的字面量必须提取为常量(局部一次性的自明字面量不强制提取)
- 组合爆炸场景(如多入口 × 多动作全排列)必须用数据驱动/结构化模型生成,禁止手写拼接 `[自检]` 最低标准: 同一模式重复 ≥5 次即应数据驱动

### 3.4 异常铁则
- 绝对禁止空 catch/except 吞异常 `[门禁]` AST 扫描
- 异常必须分类处理,保留完整栈与上下文 `[自检]` 最低标准: 每个 except 块必须有日志或 raise
- 捕获异常必须绑定变量(`as e`)且真正使用,禁止 `_ = e` 占位 `[门禁]` AST 扫描
- 异常处理块内禁止裸 `return None/False/0`(无任何日志或说明) `[门禁]` AST 扫描。豁免: 行内标记 `allow-silent-return`(具体注释语法由项目按语言约定)仅为识别标记,**必须同时登记在项目例外登记(4.4)中且经人类审批后方可生效**,缺任一条件即视为违规; 仅限同时满足**无 I/O、无状态变更、无业务分支、无外部依赖**四条件的纯转换函数(如 `safe_to_int`)
- 异常后的返回值是否携带充分上下文(日志、错误码、契约说明) `[自检]` 注: 返回结构化错误类型(Result/Option/ErrorCode)不视为裸 return,但必须可判读、可追踪
- 日志分级合规,禁止打印敏感信息(密钥、token、密码) `[自检]` 最低标准: 日志中不得出现 key/secret/password 的真实值

### 3.5 注释规则
- 仅写「为什么这么做」(业务背景、边界逻辑、踩坑记录),禁止废话注释 `[自检]` 判定标准: 注释**仅描述代码动作**且不提供原因/边界/风险信息时判定为废话; 动作词(导入/定义/调用等)仅为辅助信号,不单独定罪
- TODO 必须标注责任人、截止日期,禁止无主 TODO `[门禁]` 正则扫描
- 禁止注释掉的代码长期留存 `[自检]` 最低标准: 注释掉的可执行代码 ≥3 行即应删除

### 3.6 依赖与状态
- 函数优先纯函数: 输入确定则输出确定,不修改全局状态 `[自检]` 最低标准: 函数不得修改入参以外的模块级变量
- 依赖通过参数显式传递,禁止隐式全局变量耦合 `[自检]` 最低标准: 函数不得读取未通过参数传入的可变全局状态
- 模块职责单一: 一个文件只做一件事,禁止"万能文件" `[自检]`

**万能文件判定标准**(必须同时满足下列任意两条):
- 文件包含 ≥3 类不同职责(如 runner + asserter + error-collector)
- 文件包含 ≥3 个无关联的公开入口函数
- 文件被 ≥3 个不同业务域调用(业务域以项目模块划分文档为依据,如支付/物流/用户)

> **判定说明**: 职责单一但被广泛复用的公共模块(如 `logger.py`、`http_client.py`)不因"被多域引用"而单独判定为万能文件。若语言/框架天然将多职责聚合(如部分 OOP 框架的 Controller),以项目文档定义为准。

### 3.7 敏感信息铁则
- 严禁硬编码密钥、token、密码等敏感信息,必须通过环境变量/密钥管理服务读取 `[门禁]` gitleaks / git-secrets / truffleHog。测试用 mock 值必须显式标注为 mock(如变量名含 `mock`/`fake`/`test`),不得与真实凭证格式相同
- 严禁提交私钥(`id_rsa`)、真实凭证文件(`.env`、`.aws/credentials`)到仓库 `[门禁]` pre-commit 文件类型拦截。公钥、脱敏测试夹具、示例证书须登记在例外登记(4.4)中经人类审批后保留

### 3.8 测试底线
- 核心路径(I/O、状态变更、协议解析)必须至少有 1 条契约测试或行为快照 `[自检]`
- 重构存量代码前,必须先确认有行为兜底,无兜底则先补 `[流程]`
- **最小兜底标准**: 输入样例 + 输出样例 + 异常样例至少各 1 条

## 四、自动化门禁(fail-closed)

### 4.1 统一门禁入口

每个项目必须提供**单一标准的门禁执行入口**(如 `pre-commit-config.yaml` 或 `make check` 或统一 selfcheck 脚本),打包所有检查项,确保本地与 CI 环境结果一致。禁止零散拼凑工具导致环境差异。

### 4.2 必须自动化卡死的门禁

以下检查必须接入统一门禁入口,fail-closed,不可绕行:

| 门禁项 | 阈值 | 参考工具 |
|--------|------|----------|
| 连续空行 | ≤2 | 自定义脚本 / ruff |
| 空行占比 | ≤25% | 自定义脚本 |
| 单文件非空行 | ≤1,200(测试 ≤2,000) | 自定义脚本 / cloc |
| 单函数非空行 | ≤80 | AST 扫描脚本 / radon |
| 单类非空行(有类语言) | ≤500 | AST 扫描脚本 |
| 空 catch/except | 0 个 | AST 扫描脚本 / ruff |
| 异常处理块内裸 return | 0 个(已登记在例外登记且代码行标注 `allow-silent-return` 的纯转换函数除外) | AST 扫描脚本 |
| 无主 TODO | 0 个 | 正则扫描 |
| 编译/语法检查 | 100% 通过 | py_compile / tsc / cargo check |
| 圈复杂度 | ≤10 | radon / flake8 / eslint |
| 配置文件行数 | ≤500(JSON/YAML) | 自定义脚本 |
| 配置文件嵌套深度 | ≤5 层 | 自定义脚本 |
| 敏感信息 | 0 个 | gitleaks / git-secrets |

### 4.3 自检必查项

以下规则无法可靠自动化为硬性门禁,**执行者必须在每次交付前自行检查**(AI 主导项目由 AI 执行,非 AI 项目由开发者执行),人类可随时抽查:

| 检查项 | 量化触发条件 | AI 易犯场景 |
|--------|-------------|-------------|
| 命名语义合规 | 名称缺少业务对象或场景限定 → 判定泛化(领域术语与职责一致则豁免) | 不同会话命名风格不一致 |
| 重复代码抽离 | 同一逻辑(≥3 行)出现在两处以上 → 必须抽离 | 跨文件/跨会话产出的隐性重复 |
| 废话注释 | 注释仅描述代码动作且不提供原因/边界/风险 → 判定废话 | 默认生成叙述性注释 |
| 注释与代码一致 | 修改代码后关联注释未同步更新 → 判定腐烂 | 改了代码没改注释 |
| 禁止过度设计 | 新增抽象层但无第二个实现者 → 判定过度设计 | 主动"预留扩展"但无实际需求 |
| 异常分类处理 | except 块无日志且无 raise → 判定不合格 | 用通用 Exception 兜底所有场景 |
| 魔法值治理 | 含义不自明、重复出现、或跨函数复用的字面量 → 必须提取为常量 | 直接在代码中写死阈值/路径/端口号 |
| 复用公共函数 | 项目已有同功能函数时必须复用 → 新建前先搜索 | 上下文窗口外的已有代码被忽视 |
| 注释掉的代码 | ≥3 行注释掉的可执行代码 → 必须删除 | "先注释掉旧代码再写新的" |

### 4.4 例外登记(长期合理例外) `[人工决策]`

> **定位**: 永远无法合规的合理例外。例外登记永不转为债务白名单,不可互相替代。  
> 本规范所有放行入口统一收口为**两本账**: 例外登记(长期合理例外) + 债务白名单(限期必修)。函数级技术豁免(如 `allow-silent-return` 标记)也必须登记在例外登记中,不得独立存在。

DSL 文件、协议定义、生成器等天然体量较大且无法合理拆分的文件,以及经审批的函数级技术豁免(如纯转换函数的静默返回),走例外登记:

| 字段 | 必填 | 说明 |
|------|------|------|
| 对象标识 | 是 | 文件路径(文件级)或 `文件路径::符号路径`(函数/类/方法级,如 `src/utils.py::SafeConverter.to_int`) |
| 例外项 | 是 | 哪条门禁豁免 |
| 理由 | 是 | 为什么无法合规(如"协议定义文件不可拆分"/"纯转换函数无需日志") |
| owner | 是 | 负责持续评估的**模块负责人** |
| 复查周期 | 是 | 如"每季度复查一次是否仍合理" |

- 例外登记不设到期清零,但必须定期复查
- 理由不再成立时必须撤销例外并治理
- 例外登记条目数超过项目约定上限时,须人工审批后方可新增
- **执行者不得自行创建例外登记,必须由人类审批**

### 4.5 债务白名单(存量违规限期清零)

> **定位**: 必须在期限内治理达标的违规。债务白名单永不转为例外登记,不可互相替代。

**增量代码**: 无例外,必须 100% 通过所有自动化门禁。

**存量违规文件**: 列入债务白名单:

| 字段 | 必填 | 说明 |
|------|------|------|
| 对象标识 | 是 | 文件路径(文件级)或 `文件路径::符号路径`(函数/类/方法级) |
| 违规项 | 是 | 哪条门禁超限 |
| 当前值 | 是 | 如"函数 collect_errors 非空行 = 13,065" |
| 目标值 | 是 | 如"≤80" |
| 责任人 | 是 | **模块 owner / 业务负责人**(AI 为执行者) |
| 到期日 | 是 | 见下方分级规则 |

**白名单仅允许可延期的结构性债务**(体量/复杂度超限)进入。以下项目**永不白名单,发现即修复**:
- 敏感信息泄露 → 立即阻断
- 空 catch/except 吞异常 → 立即阻断
- 并发写入覆盖 → 立即阻断
- 编译/语法错误 → 立即阻断
- 无主 TODO → 修复成本极低(加责任人+日期即可),直接修复,不进白名单

**严重度与到期日**(适用于可白名单的结构性债务):
- P0(超限 ≥10 倍)≤1 周, P1(超限 2-10 倍)≤1 个月, P2(轻微超限)≤3 个月

**管控规则**:
- 白名单内的文件: 门禁允许通过但打印警告
- 到期未治理: 门禁直接失败,不再豁免
- 每完成一个文件的治理就从白名单移除
- 白名单条目数超过项目约定上限时,须先治理旧条目才能新增

**紧急 Bypass 机制**: 仅适用于**已登记的结构性债务逾期**与线上 P0 故障/硬性发版冲突时。不得用于敏感信息、空 catch、编译错误、测试失败等即时高风险项。执行条件:
- 由项目负责人(人类)审批,在 commit message 中标注 `[BYPASS-DEBT: 原因]`
- 指定新到期日(≤1 周)
- 同一条债务最多 Bypass 一次,再次触发必须启动专项治理
- Bypass 记录纳入项目审计存档,下次健康审计必须复查

### 4.6 生成链路规范
- 所有自动生成的代码文件,生成后必须经过 formatter(如 black / ruff format / prettier)再写入
- 禁止"生成 → 直接写盘"的裸链路,这是格式污染的第一根因
- 生成链路变更后,应对比前后输出差异,确认无意外膨胀。膨胀阈值可按文件类型配置(建议: 业务代码 >20%、测试 >30%、协议/配置 >10%; 未配置时默认 >20%),超出则人工复查

## 五、存量屎山治理流程

当发现存量代码违反本规范时,严格按以下流程执行,禁止跳步:

### 5.1 治理总原则
1. **先堵增量,再治存量** — 门禁永远比重构优先
2. **先锁行为,再动结构** — 拆分前必须有行为兜底(测试或快照)
3. **先去污,再拆分** — 格式归一化是零风险的第一刀
4. **小步灰度,随时回滚** — 拆一组验一组,绝不全拆再祈祷
5. **按职责拆文件,不只拆函数** — 目标是"职责单一的小模块",不是"行数合规的大文件"
6. **新功能严禁直接在屎山上堆砌** — 必须通过防腐层隔离: 新建独立 Adapter 文件,旧系统输入在 Adapter 内转换为新数据模型,新逻辑不直接 import 违规旧模块
7. **发现即上报** — 开发过程中发现存量违规,必须立即上报,不得"先忍着后面再说"

### 5.2 治理执行步骤

```
Step 1  诊断 ─── 扫描全仓,列出所有违规文件/函数/类,标注严重度,报告人类
                  结构性: P0 = 超限≥10倍  P1 = 超限2-10倍  P2 = 轻微超限
                  即时修复项(敏感信息/空catch/并发覆盖/编译错误/无主TODO): 不进白名单,直接修复
        │
Step 2  堵增量 ── 部署自动化门禁(第四章),结构性存量走债务白名单,新增代码立即生效
        │
Step 3  锁行为 ── 为即将拆分的模块建立行为兜底:
                  优先: 补契约测试(输入→输出→异常 三要素)
                  兜底: 录制当前 CLI 输出 / JSON 快照 / 关键路径日志
                  最小兜底标准: 输入样例+输出样例+异常样例至少各1条
                  后续拆分只要行为兜底全绿即证明逻辑无变更
        │
Step 4  去污 ─── 格式归一化(空行压缩、formatter),零逻辑改动,零风险
                  验证: 编译通过 + 全量测试通过
                  格式化改动单独提交,不混入业务变更
        │
Step 5  拆分 ─── 按职责拆文件,每拆一组形成绿色检查点(commit 或 patch snapshot)
                  回归全绿才继续,入口文件最终只保留 import + 串联
        │
Step 6  验收 ─── 全量回归通过 + 从债务白名单移除 + 所有指标达标
                  向人类报告治理结果
```

### 5.3 拆分原则与判定标准

**职责判定**: 输入源、处理逻辑、输出目标三者一致的代码归为同一职责。三者中任意一个不同,就应拆到不同模块。

**职责拆分 checklist**(拆分前逐项检查):
- [ ] 是否同时处理 I/O 和业务逻辑 → 应拆为 I/O 层 + 逻辑层
- [ ] 是否同时包含命令执行和结果断言 → 应拆为 runner + assertions
- [ ] 是否同时包含数据解析和错误收集 → 应拆为 parser + errors
- [ ] 拆分前列出模块依赖关系,拆分后不得增加循环依赖
- [ ] 拆分完成后重新确认依赖关系,验证无新增循环依赖

> **示例**: 一个 10,000 行函数同时做"启动子进程运行命令" + "解析 JSON 断言字段" + "收集错误生成报告" → 应拆为 `runner.py`(子进程) + `assertions.py`(断言) + `errors.py`(报告),各自独立可测。

- 按职责拆,不按行数拆
- 提取过程中不修改任何判断逻辑,只做"搬运"
- 若任何一步回归不绿,立即停止,回滚到上一个绿色检查点
- 拆分完成后,原入口文件只做 import + 顶层串联

### 5.4 违规处理流程

```
发现违规 →
    │
    ├─ 即时修复项(敏感信息/空catch/并发覆盖/编译错误/无主TODO): 立即修复,不进白名单
    │
    ├─ 增量结构性违规: 提交被拒绝,执行者自行修复后重新提交
    │
    └─ 存量结构性违规: 上报人类 → 录入债务白名单(含责任人+到期日)
              → 按计划执行治理 → 完成后从白名单移除
              → 到期未完成 → 门禁升级为失败(紧急情况见 Bypass 机制)
```

## 六、长效免疫(防止复发)

### 6.1 提交纪律
- 业务改动: 单次提交建议 ≤500 行 `[流程]`
- 格式化/重命名/批量生成等机械改动: 单独提交,不计入上述限制 `[流程]`
- 超大提交必须在 commit message 说明原因 `[流程]`
- 每次提交前必须过自动化门禁 `[流程]`

### 6.2 定期健康审计
- 每完成一个里程碑,跑一次全仓健康扫描并向人类报告
- 扫描指标: 文件非空行 / 函数非空行 / 空行比 / 最大连续空行 / 圈复杂度
- 新增违规立即治理,不累积
- 与上次扫描对比,确认趋势向好而非恶化
- 复查所有 Bypass 记录,确认已完成补治理

### 6.3 数据驱动替代手写(中期目标)
对组合爆炸类场景(如测试矩阵、多入口覆盖),逐步引入结构化模型:

```python
Case(
    executor="root-ps1",
    action="seed-failed",
    task_id="task-001",
    expect_exit=0,
    expect_fields=["status", "task_id"],
)
```

由模型自动生成命令 + 断言,新增场景只加一条定义,不碰逻辑代码。

## 七、屎山类型速查表

| 类型 | 症状 | 检测 | 预防/治理 |
|------|------|------|-----------|
| 格式污染 | 空行占比 >25%、连续空行 >2 | `[门禁]` | `ruff format` / `black` 一键修复 + 门禁 |
| 函数膨胀 | 单函数 >80 行 | `[门禁]` | 门禁 + 按职责拆函数/拆文件 |
| 模块失控 | 单文件 >1,200 行 | `[门禁]` | 门禁 + 按职责拆文件 |
| 万能文件 | 满足判定标准任意两条(3.6) | `[自检]` | 按职责拆文件 |
| 组合爆炸 | 手写排列组合,行数指数增长 | `[自检]` | 结构化模型,数据与逻辑分离 |
| 复制粘贴 | 同一逻辑出现在两处以上 | `[自检]` | 抽离公共函数 |
| 全局状态 | 函数依赖/修改外部变量 | `[自检]` | 显式入参,纯函数优先 |
| 空 catch 吞异常 | `except: pass` / `catch {}` | `[门禁]` | 发现即阻断,补异常日志与分类处理 |
| 静默降级 | 异常处理块内裸 return(非纯工具函数) | `[门禁]` | 补日志/上下文,或改用显式错误协议 |
| 无主 TODO | TODO 无责任人/截止日 | `[门禁]` | 发现即修复(加责任人+日期) |
| 无测试裸奔 | 核心路径无测试/快照 | `[自检]` | 重构前必须先有行为兜底 |
| 过度设计 | 新增抽象层但无第二个实现者 | `[自检]` | KISS 优先,必要才抽象 |
| 注释腐烂 | 注释描述的行为与代码不一致 | `[自检]` | 只写"为什么",禁止废话注释 |
| 配置爆炸 | JSON/YAML >500 行或嵌套 >5 层 | `[门禁]` | schema 校验 + 配置拆分 |
| 敏感信息泄露 | 硬编码密钥/token/密码 | `[门禁]` | 发现即阻断,环境变量 + gitleaks |

## 八、绝对红线禁令(触碰即作废,零例外)

1. 严禁私自修改/新增锁定的目录结构、依赖白名单、架构核心文件
2. 严禁跨目录乱放文件、测试文件乱放乱命名
3. 严禁使用无意义泛化命名(名称缺少业务对象或场景限定)
4. 严禁复制粘贴开发、空 catch 吞异常、魔法值硬编码
5. 严禁超审批范围开发、修改非指定边界内的文件
6. 严禁输出未通过自动化门禁的代码
7. 严禁无测试/无快照覆盖盲目重构存量核心代码
8. 严禁在存量屎山上直接堆砌新功能(必须防腐层隔离)
9. 严禁"生成 → 直接写盘"的裸链路(必须经过 formatter)
10. 可延期的存量结构性违规必须录入债务白名单并明确到期日; 即时修复项(无主 TODO、空 catch 等)发现即修,不得进入白名单; 新增代码不得引入任何未整改的新增违规
11. 严禁硬编码密钥、token、密码等敏感信息
12. 严禁提交私钥、真实凭证文件到仓库(公钥/测试夹具须登记在例外登记中)

## 九、AI 主导开发模式

> 本章在项目采用 AI 主导开发(代码主要由 AI 产出,人类负责方向与决策)时生效。

### 9.1 AI 执行主序列(每次改动必须走完)

```
修改前: 搜索项目已有公共函数 → 同步远端基线(具体方式由项目选定) → 扫描当前状态记录违规基线
   ↓
修改中: 遵守所有红线生成代码,优先复用已有模块
   ↓
修改后: 运行全部自动化门禁 + 按 P0→P1→P2 执行自检清单 → 对比修改前后
   ↓
  ├─ 全绿 + 自检通过 + 无新增违规 → 输出自检结论 → 交付
  ├─ 门禁不绿 → 输出: 违规项+当前值+红线值+修复计划 → 自行修复 → 循环(至多3次)
  │              → 3次仍未修复: 熔断挂起,上报人类,不得继续盲改
  ├─ 自检发现问题 → 自行整改 → 重新检查
  └─ 发现存量违规 → 上报人类(违规项+当前值+红线值)
```

### 9.2 角色分工

| 角色 | 职责 | 决策权 |
|------|------|--------|
| 人类 | 定需求方向、审批架构变更、拍板例外/白名单、抽查代码质量 | 所有 `[人工决策]` 事项 |
| AI | 编码实现、执行门禁、自检 `[自检]` 项、上报违规、执行治理 | 所有 `[门禁]` + `[自检]` 事项 |

**核心约束**: AI 不得替人类做决策(如变更架构、新增依赖、突破需求边界); 人类不应替 AI 做检查(门禁和自检由 AI 全权负责)。

### 9.3 质量约束
1. AI 输出的代码执行本规范全部标准,不因"AI 写的"而降低要求
2. AI 禁止生成超过门禁阈值的函数/类/文件,必须在生成时就拆分到位
3. AI 重构存量代码前,必须先确认有行为兜底(测试或快照),无兜底则先补
4. AI 必须主动搜索项目已有的公共函数/模块,优先复用而非重新生成

### 9.4 并发写入纪律
5. **单写者规则**: 同一仓同一分支同一时刻只允许一个 AI 实例写入 `[流程]`
6. 并行开发必须拆到独立 worktree 或 branch `[流程]`
7. 接手前必须同步远端基线并核对未提交改动归属(具体同步方式由项目选定); 存在非 AI 的未合并改动时暂停并上报人类; 人类紧急修改后,AI 必须重新同步 + 全量门禁 + 自检后才能继续写入 `[流程]`
8. 切换窗口或等待额度恢复前,先做一次绿色检查点(commit 或 stash) `[流程]`

### 9.5 AI 高频踩坑清单

| 问题 | 根因 | 防范措施 |
|------|------|----------|
| 跨会话隐性重复 | 不同会话的 AI 无法感知彼此产出 | 每次改动前全局搜索同类函数 |
| 过度设计 | AI 倾向"预留扩展" | 无第二个实现者不得新增抽象层 |
| 废话注释 | AI 默认生成叙述性注释 | 仅描述动作且无原因/边界的注释即删 |
| 忽视已有公共模块 | 上下文窗口外的代码被忽视 | 修改前搜索项目已有工具函数 |
| 命名风格不一致 | 不同会话生成不同风格 | 按项目已有命名惯例统一 |
| 注释未同步更新 | 改了代码没改注释 | 改代码后必须检查关联注释 |
| 一次性大量生成 | AI 不知疲倦,容易一次产出巨量代码 | 遵守提交纪律,分步交付 |

### AI 自检清单(每次交付前按 P0→P1→P2 顺序执行,不得跳级)

**P0 — 必须首先检查(影响正确性)**:
- [ ] 所有自动化门禁是否通过
- [ ] 是否有新增违规(对比修改前后)
- [ ] 是否确认当前是唯一写入者
- [ ] 是否有硬编码的密钥/token/密码

**P1 — 必须检查(影响质量)**:
- [ ] 是否在存量屎山上直接堆砌了新功能
- [ ] 是否有未经 formatter 的生成文件
- [ ] 是否有复制粘贴的重复代码(全局搜索确认)
- [ ] 是否复用了项目已有的公共函数(而非重新生成)
- [ ] 是否有魔法值硬编码(应提取为常量)

**P2 — 必须检查(影响规范性)**:
- [ ] 是否有注释掉的代码残留(应删除)
- [ ] 是否有废话注释(仅描述动作且无原因/边界即删)
- [ ] 是否有改了代码但未更新的注释(应同步)
- [ ] 4.3 自检表各项是否逐项确认

