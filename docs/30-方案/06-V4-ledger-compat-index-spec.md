# V4 台账兼容索引方案（Slice A）

## 目的

- 本文件定义三份主台账在迁移前后的兼容索引规则。
- 当前只定义新旧路径映射、读取优先级、回退顺序与写入规则。
- 本轮不迁移任何真实文件，不改历史数据。

## 适用对象

- `开发计划.md`
- `MVP_PROGRESS.md`
- `PUSH_LOG.md`

## 目标

### 一、给三份台账建立统一索引入口
- 不再让脚本和文档各自硬编码根目录路径。
- 后续迁移时，优先改索引，不直接改每个调用点。

### 二、允许迁移期双路径并存
- 旧路径短期继续可读。
- 新路径准备好后，读取逻辑按优先级回退。
- 真正切换后，再逐步收旧路径。

## 规划中的目标路径

- `docs/records/开发计划.md`
- `docs/records/MVP_PROGRESS.md`
- `docs/records/PUSH_LOG.md`

> 说明：本轮只定义这些目标路径，不创建 `docs/records/`。

## 兼容索引规则

### 一、索引对象
每一份台账都应有以下信息：
- `logical_id`：逻辑标识
- `legacy_path`：当前旧路径
- `target_path`：迁移目标路径
- `read_order`：读取优先级
- `write_mode`：当前阶段允许的写入规则
- `cutover_state`：切换状态

### 二、逻辑标识
- `dev-plan`
- `mvp-progress`
- `push-log`

### 三、旧路径与目标路径映射
- `dev-plan`：`开发计划.md` -> `docs/records/开发计划.md`
- `mvp-progress`：`MVP_PROGRESS.md` -> `docs/records/MVP_PROGRESS.md`
- `push-log`：`PUSH_LOG.md` -> `docs/records/PUSH_LOG.md`

## 读取优先级

### Phase A：当前阶段
- 只读旧路径
- 新路径尚未创建
- 任何脚本不得假设新路径已存在

### Phase B：兼容阶段
- 优先读新路径
- 新路径不存在时回退读旧路径
- 若新旧路径同时存在且内容不一致，直接报错，不允许静默取其一

### Phase C：切换完成
- 只读新路径
- 旧路径只允许保留跳转说明或迁移标记，不再作为真实数据源

## 写入规则

### 当前规则
- 仍写旧路径
- 禁止在未完成切换前双写，避免产生双份真源

### 未来切换规则
- 当 `cutover_state=target-primary` 后，写入目标路径
- 旧路径仅保留显式说明，不再写真实内容

## 切换状态定义

- `legacy-only`：仅旧路径存在
- `dual-readable`：新旧路径都可读，但以新路径优先
- `target-primary`：新路径为唯一真源
- `legacy-retired`：旧路径只剩跳转说明或已移除

## 冲突处理规则

### 一、同时存在且内容不一致
- 立即失败
- 不允许自动合并
- 不允许静默覆盖

### 二、目标路径缺失
- 在 `legacy-only` 和 `dual-readable` 阶段允许回退旧路径
- 在 `target-primary` 阶段视为错误

### 三、索引缺项
- 缺少 `logical_id / legacy_path / target_path / cutover_state` 视为配置错误

## 推荐落地顺序

1. 先把这份兼容索引规则固定下来。
2. 再让检查脚本支持“按索引找台账”，而不是直接写死根路径。
3. 然后先迁 `开发计划.md`，再迁 `MVP_PROGRESS.md`，最后迁 `PUSH_LOG.md`。

## 当前完成判据

- 三份台账的新旧路径映射已明确。
- 读取优先级、写入规则、切换状态、冲突处理已明确。
- 后续脚本改造可以直接引用本方案，不再重新拍脑袋定规则。
